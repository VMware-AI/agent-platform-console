# CLAUDE.md — 开发框架(团队约定)

> 本仓库(agent-platform-console)AI 辅助开发的团队约定。**后续开发,所有人 / agent 一律 follow 本框架**,直到维护者宣布进入收尾/发版阶段。

## 1. 测试:开发期「不反复维护」

项目仍在快速迭代、远未收尾。为控制成本与保持开发速度,**开发阶段不反复维护/运行单元测试**:

- ❌ **改源码时不要顺手改动或新增对应测试文件**(`*.test.ts` / `*.spec.ts`)。源码变更不应连带 diff 测试文件。
- ❌ **不主动跑全套 vitest**(本仓已有 400+ 用例)。单次改动不跑全套,不为一次改动读一屏测试输出。
- ✅ **CI 是安全网**:合并前依赖 CI 的构建/类型/lint gate 兜底,不在本地反复跑测试。**PR/push 上根本不跑 test**(省 CI 资源,谁都不因单测被卡);测试由独立的 `test.yml` **每周定时(周一)+ 手动 `workflow_dispatch`** 触发,只测 main。
- ✅ **测试文件保留、不删**——只是暂不主动维护,供每周测试轮次使用。
- 🗓️ **每周一次**:`test.yml` 定时跑出结果(或维护者手动 dispatch),维护者据此统一审核;**此时**才更新/补齐/修复测试。
- **唯二例外**:①被明确要求写/跑测试;②处于维护者主导的每周测试轮次内。

> 恢复常规 TDD / 覆盖率要求的时机:项目进入收尾或发版阶段,**由维护者显式宣布**,届时更新本文件。

## 2. CI gate 不得绕过

- **gate = `ci.yml` 的 `Lint, type-check & build` job**(不含 test)。gate 红就修,不得 admin override 强合 / 绕过;合并前确认它全绿即可。测试不在合并路径上,`test.yml` 的结果不影响合并。
- 绕过 gate 会让主干进入「gate 重跑即红」的漂移态,任何基于它的新 PR 都被动继承、踩坑(近期 backend 已有此类教训)。

## 3. 范围最小化

- 一次 PR 聚焦一个模块 / 一件事;不夹带无关重构或大范围格式化。
- 工具/版本差异(如本机 vs CI 的 gofmt/node 版本)导致的无关 diff,先对齐 CI 版本再改,避免污染 PR。

---
*本框架为临时开发期约定,优先「开发速度 + 控成本」;质量兜底交给 CI gate + 维护者每周轮次。*
